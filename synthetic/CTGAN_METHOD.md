# Synthetic Data Augmentation — Method, Rationale & Bias (for the report)

> Generated by `synthetic/train_ctgan.py`. This document satisfies the brief §4 /
> checklist item *"synthetic method documented in report (method, rationale, bias/limits)"*.

## 1. Method
- **Model:** Conditional Tabular GAN — SDV `CTGANSynthesizer` (`sdv.single_table`).
- **Training data:** the original UCI *Default of Credit Card Clients* table (~30,000 rows),
  loaded from `raw/uci_credit_card.xls` (or the `.csv` fallback).
- **Metadata:** auto-detected with `SingleTableMetadata`, then `SEX`, `EDUCATION`, `MARRIAGE`
  and the target (`default payment next month`) are forced to **categorical** so CTGAN models them
  as discrete and so we can sample conditionally on the target. `ID` is marked as an `id` column and
  dropped before training (surrogate IDs are re-assigned after generation).
- **Generation volume:** **≥150,000** synthetic clients (`--n 150000`), comfortably above the
  ≥100,000-row brief requirement. After the ETL unpivot to monthly grain (×6 months) this becomes
  ≈900,000 fact rows.
- **Target-class control:** generation uses **conditional sampling** (`sdv.sampling.Condition`) so the
  default/non-default split is *deterministic*, not left to the generator:
  - default mode **preserves the original class ratio** (~22% default),
  - `--balance` forces an even **~50/50** split for class-balanced model training downstream.
- **Reproducibility:** a single `--seed` (default 42) seeds Python `random`, NumPy and (if present)
  PyTorch (`manual_seed` + `cuda.manual_seed_all`) before `fit`.

## 2. Rationale (why CTGAN)
- The credit features are **mixed-type** (continuous `LIMIT_BAL`/`BILL_AMT*`/`PAY_AMT*`, categorical
  demographics, ordinal `PAY_*` status). CTGAN's mode-specific normalisation + conditional generator
  handle mixed tabular data and **multimodal / imbalanced** columns better than Gaussian-copula or
  naïve resampling.
- It lets us hit the big-data row target **without collecting more real PII**, and the conditional
  sampler gives explicit control over the default class — important because default risk is the label
  the whole project analyses.

## 3. Bias & limitations (must be acknowledged)
- **Fidelity ≤ source.** CTGAN can only reproduce patterns present in the ~30k seed rows; rare
  segments (very high limits, unusual payment sequences) are under-represented or smoothed.
- **Correlation drift.** GAN-synthesised columns may weaken real cross-column dependencies
  (e.g. `BILL_AMT` ↔ `PAY_AMT` ↔ default), which can bias downstream feature/correlation analysis.
- **Amplified source bias.** The 2005 Taiwan sample's demographic skew is *learned and amplified* at
  150k scale — synthetic results are **not** representative of any current/other population.
- **Balanced mode caveat.** `--balance` (50/50) deviates from the real ~22% prevalence; it aids model
  training but **inflates the apparent default rate**, so dashboard "overall default rate" KPIs must
  state whether they run on real, preserve-ratio, or balanced data.
- **No new signal.** Synthetic rows add volume, not information; they must not be presented as
  independent real-world evidence.
- **Validation.** Optionally quantify fidelity with SDV's `evaluate_quality` / diagnostic reports and
  cite the score in the report.

## 4. Reproduce
```bash
python synthetic/train_ctgan.py --n 150000 --epochs 300 --seed 42        # preserve ~22% default
python synthetic/train_ctgan.py --n 150000 --epochs 300 --seed 42 --balance  # ~50/50
```
Run parameters and the realised class balance are written to `synthetic/ctgan_summary.json`.
