## 1. What each pipeline actually does here
| | **ETL** (Tableau) | **ELT** (Power BI) |
|---|---|---|
| Where transform runs | Spark **outside** the warehouse (PySpark DataFrame API) | **Inside** Hive (set-based SQL) |
| Style | Row-level: clean → unpivot → engineer features → validate | Set-based: `stack()` unpivot, **window funcs**, **GROUPING SETS** OLAP |
| Output | `bigdata_etl` star schema (`fact_credit_monthly` + dims) | `bigdata_elt` materialised analytical tables |
| Code | `etl_pipeline/transform.py`, `load.py` | `elt_pipeline/transform.sql` + thin runners |

## 3. Comparison dimensions
| Dimension | ETL | ELT | Notes |
|---|---|---|---|
| **Runtime profile** | Heavier transform stage (Python/Spark compute before load) | Light load, heavier in-warehouse SQL | see §2 numbers |
| **Resource usage** | Spark executor memory during transform | Warehouse/SQL engine does the work | peak_rss from metrics |
| **Code complexity (LOC)** | _from summary_ | _from summary_ | proxy only; SQL is denser per line |
| **Where the transform load sits** | Upstream (pre-load) | Downstream (post-load, in Hive) | core ETL/ELT distinction |
| **Flexibility** | Full Python ecosystem (ML features, libs) | SQL-bounded but instantly re-runnable on loaded data | |
| **Data freshness** | Data usable only **after** transform completes | Raw queryable **immediately** after load; transforms refresh on demand | ELT favours fast landing |
| **Schema-on** | Schema-on-write (clean, modelled star) | Schema-on-read then materialise | |
| **Reprocessing** | Re-run Spark job end-to-end | Re-run SQL over already-loaded raw (cheaper iteration) | |

## 4. Suitability for this use case
- **ETL** wins where the analysis needs **engineered/ML features** and a clean, governed star
  schema (our `credit_utilization`, `payment_ratio`, delay features, validation rules).
- **ELT** wins for **fast landing + flexible in-warehouse exploration** (window/OLAP) and cheap
  iteration once data is in Hive.
- **For this project:** ETL gives the higher-quality modelled layer for the default-risk features;
  ELT demonstrates set-based analytics and quicker raw availability. Using both lets the report
  compare them directly — and the Tableau vs Power BI dashboards (built on the two outputs) are the
  visual evidence (bonus, §9).
